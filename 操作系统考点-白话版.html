1 操作系统概述

🔄 并发多程序"同时"跑。宏观同时，微观交替。单核轮流执行
🤝 共享多进程共用系统资源。互斥共享（打印机）或同时共享（读文件）
🎭 虚拟一个物理实体映射为多个逻辑物。虚拟内存让每进程以为独占全部内存
⚡ 异步进程走走停停，推进速度不可预知
并·共·虚·异 — "病假工虚，一同进医院" （并发·共享·虚拟·异步）
并发 ≠ 并行！并发是逻辑上的"同时"（单核交替），并行是物理上的同时（多核真正一起跑）。
发展阶段特点
手工操作人直接操作面板，纸带打孔
单道批处理攒一批任务一起跑，不能交互
多道批处理内存同时放多道程序，CPU 和 I/O 可并行
分时系统多人共用一台主机，每人轮流用时间片，你感觉独占
实时系统必须在规定时间内响应（导弹、工控）

系统调用 是 用户态→内核态的唯一途径。用户程序通过 int 0x80（x86）或 syscall（x64）指令陷入内核。

2 进程管理

🔷 进程 vs 线程

进程 = 工厂
• 拥有独立资源
• 切换代价大
• 通信靠 IPC（管道、消息队列）
线程 = 工人
• 共享进程的资源
• 切换轻量
• 直接读写共享变量

🔷 进程状态切换

创建→ 就绪→ 运行→ 终止
就绪→运行：被调度选中 | 运行→就绪：时间片用完 | 运行→阻塞：等 I/O、等锁 | 阻塞→就绪：I/O 完成
三态循环：就跑等 — "就跑着等"（就绪→运行→阻塞→就绪循环）

🔷 临界区与锁

访问共享资源的代码段 = 临界区。必须保证同一时刻只有一个进程在里面。

访问准则：空闲让进 · 忙则等待 · 有限等待 · 让权等待

3 CPU 调度算法

核心问题：CPU 只有一个，几十个进程等着。让谁先跑？跑多久？
算法一句话抢占？
FCFS谁先到谁先跑。简单但长进程堵死短进程否
SJF最短的先跑。平均等待最少，但长进程可能饿死均可
HRRN ★响应比 = (等待+服务)÷服务。综合公平否
RR每人跑一个时间片 q，轮流切换是
多级反馈队列 ★★多级队列，高优先时间片短，没跑完降级是
响应比 = (等待时间 + 服务时间) ÷ 服务时间

等待越久、需求越少 → 响应比越高 → 越优先。兼顾长短作业。

周转时间 = 完成时间 − 到达时间
带权周转时间 = 周转时间 ÷ 服务时间
考试套路：给进程表 → 画甘特图（时间轴） → 算每个的周转时间 → 求平均

4 PV 操作 — 大题必考 ⭐⭐⭐

🔷 信号量 = 计数器 + 排队队列

P 操作S = S−1；如果 S<0，排队等着
申请资源，可能会阻塞
V 操作S = S+1；如果 S≤0，叫醒队里一个人
释放资源，不会阻塞

🅿️ = 请求（Please） | ✅ = 释放（Vacate）

🔷 生产者-消费者 ★★

// 货架有 N 个空位，初值 empty=N, full=0, mutex=1

生产者：
  P(empty);    // ① 先看有没有空位
  P(mutex);    // ② 占货架
  放面包;
  V(mutex);    // ③ 放好，走人
  V(full);     // ④ 面包+1，有顾客等就叫醒

消费者：
  P(full);     // ① 先看有没有面包
  P(mutex);    // ② 占货架
  取面包;
  V(mutex);    // ③ 取好，走人
  V(empty);    // ④ 空位+1，有面包师等就叫醒
P(资源) 必须在 P(mutex) 前面！先拿锁再等资源 → 你拿着锁睡觉，别人想释放资源但拿不到锁 → 死锁

🔷 读者-写者 ★★

// rw=1（公告栏锁）, mutex=1（保护读者计数）, readcount=0

读者：
  P(mutex);
  readcount++;  if(readcount==1) P(rw);   // 第一个读者锁住写者
  V(mutex);
  // 读公告...
  P(mutex);
  readcount--;  if(readcount==0) V(rw);   // 最后一个读者释放
  V(mutex);

写者：
  P(rw);              // 等所有人（读者和写者）走
  // 写公告...
  V(rw);
一读锁，末读放。写者独占等全部。

🔷 哲学家就餐 ★

五个人围圆桌，五根筷子。每人需要左右两根才能吃。

每人先拿左边再等右边 → 全死锁

解法：最多允许 4 人同时拿筷子（信号量限制），或奇数号先左后右、偶数号先右后左。

5 死锁

🔷 四个必要条件（简答必背！）

① 互斥资源一次只能给一个人用（厕所一次进一人）
② 请求并保持拿着已有的，还要申请新的（占厕所等洗衣机）
③ 不可抢占不能硬抢别人的资源（不能把人从厕所踹出来）
④ 循环等待A 等 B、B 等 C、C 等 A，形成环
互·请·不·循 — "护·请·不·巡" （互斥、请求保持、不可抢、循环等）
破坏任一条件就能防死锁。比如：允许抢占（破③）、信号量做限制不让所有人都申请（破④）。

🔷 银行家算法

每次分配资源前先判断：分完之后系统还在"安全状态"吗？能找到全部分配顺序 = 安全，找不到 = 拒绝分配。

考题：给表（进程/已分配/还需/可用），判断当前状态是否安全，某进程的请求能否满足。

6 内存管理

🔷 虚拟内存

每个进程"以为"自己独占全部内存。真实物理内存 + 硬盘交换区 = 虚拟地址空间。OS 偷偷把虚拟地址翻译成物理地址。

🔷 分页 = 把内存切成固定大小的块

  • 虚拟地址侧叫 页（page）
  • 物理地址侧叫 块/页框（frame）
  • 大小通常 4KB（= 4096 = 2^12）

🔷 页表 = 翻译字典：页号 → 块号

虚拟地址 = [ 页号 | 页内偏移（低 12 位）]

↓ 查页表

物理地址 = [ 块号 | 页内偏移（直接复制）]
例：页大小 4KB，虚地址 0x5A38
高 4 位 = 5（页号），低 12 位 = A38（偏移不翻译）
页表：页 5 → 块 7
→ 物理地址 = 0x7A38

TLB（快表）：CPU 里的小缓存，存最近用过的几个页表项。命中率 90%+，省去查内存中页表的时间。
多级页表：页表本身也很大，把页表也分页，只把用到的部分放内存。

7 页面置换

核心问题：物理内存满了，要把一页换出去。换谁？
算法赶走谁特点
OPT将来最久才用到的🏆 最优但做不到（不知道未来），只当基准
FIFO最先进来的简单但有 Belady 异常（页越多反而缺页越多！）
LRU ★过去最久没被用过的效果好，"用进废退"，实现代价大
CLOCK ★扫一圈，把没被访问过的赶走近似 LRU，实现简单，实际系统用
改进 CLOCKA=0,M=0 → A=0,M=1 → A=1,M=0 → A=1,M=1优先赶走未修改的（省去写回磁盘）
FIFO 最简单但有怪病（Belady）。LRU 合理但费钱。CLOCK 折中最好用。
考题：给引用串 + N 个框 → 一步步画 → 标缺页 → 数缺页次数和缺页率

缺页中断：要访问的页不在内存（在硬盘），OS 从磁盘调入。
抖动（Thrashing）：分给进程的页太少，频繁缺页，时间全花在换页上。你桌面只剩巴掌大，写一行代码就要起身去书架 → 一上午啥也没干。

8 磁盘调度

磁头要在盘面上来回移动。按什么顺序处理请求 → 总移动距离最短？
算法一句话
FCFS按请求顺序来。公平但移动多
SSTF离当前磁头最近先去。快但远处可能饿死
SCAN ★电梯算法：一个方向走到底，路上有请求就处理，到底折返
C-SCAN ★循环扫描：只在一个方向处理，到底直接跳回起点（不处理）
电梯去程（←）接待，回程（→）不干活 = LOOK。实际系统最常用。
考题：给磁头位置 + 请求序列 → 画移动轨迹 → 算总寻道长度

9 文件系统

📋 inode文件的"身份证"。存大小、位置、权限。不存文件名！
📁 目录本质是一张表：文件名 → inode 号
📦 超级块整个文件系统的元信息（总大小、空闲块数）
🧱 块最小读写单位（通常 4KB）

🔷 路径解析：/home/xiao/file.txt 怎么找？

查 / 的 inode→ 找到 home 的 inode→ 找到 xiao 的 inode→ 找到 file.txt 的 inode→ 读取文件

🔷 硬链接 vs 软链接

硬链接软链接（符号链接）
本质多个文件名 → 同一个 inode一个文件存目标路径字符串
删原文件✅ 不影响，其他硬链接还能访问❌ 失效（悬空）
跨文件系统❌ 不能✅ 可以
硬链接是复印文件身份证，软链接是留便条"他在某某地方"。

🔷 文件分配方式

方式原理问题
连续分配文件占连续磁盘块外部碎片
链接分配每块存下一块的指针随机访问慢
索引分配 ★用一个索引块集中存所有数据块地址Unix/Linux 用这个

10 I/O 管理

🔷 控制方式演进（越来越省 CPU）

程序轮询→ 中断驱动→ DMA→ 通道
方式CPU 干什么
轮询CPU 不停检查"好了没？好了没？" — 全浪费
中断CPU 布置任务后干别的，外设好了发中断通知
DMA ★CPU 只下命令，DMA 控制器自己搬数据，搬完通知 CPU
通道更高级的 DMA，有自己的指令集

🔷 缓冲

CPU 太快，I/O 太慢 → 缓冲当中转站平滑速度差。
单缓冲 → 双缓冲（交替用，并行度↑）→ 环形缓冲（生产者消费者模型）

🔷 SPOOLing（假脱机）

多用户"同时"用打印机 → 实际 OS 先把任务存磁盘，再逐个发给打印机。用户以为自己独占打印机。
