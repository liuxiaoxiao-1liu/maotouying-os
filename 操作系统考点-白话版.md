## 一、操作系统概述

### 1.1 操作系统是什么

OS 是硬件和应用程序之间的中间人。你不需要直接操作 CPU、内存、硬盘，OS 帮你管。

**三个核心职责**：管 CPU（进程调度）、管内存（虚拟内存）、管文件（文件系统）。

### 1.2 OS 四大特征（简答题高频）

| 特征 | 一句话 |
|------|--------|
| **并发** | 多个程序"同时"运行。宏观同时，微观交替（单核轮流跑）|
| **共享** | 多个进程共用系统资源。互斥共享（打印机）或同时共享（读文件）|
| **虚拟** | 一个物理实体映射为多个逻辑对应物。虚拟内存让每个进程以为独占全部内存 |
| **异步** | 进程走走停停，推进速度不可预知 |

> 并发 ≠ 并行。并发是逻辑上的"同时"（单核交替），并行是物理上的同时（多核真正一起跑）。

### 1.3 OS 发展史

```
手工操作 → 单道批处理 → 多道批处理 → 分时系统 → 实时系统 → 网络/分布式
```

- **批处理**：一堆任务攒着一起跑，用户不能交互
- **分时系统**：多个终端共用一台主机，每人轮流用一小段时间片，你感觉在独占
- **实时系统**：必须在规定时间内响应（导弹、工业控制）

### 1.4 系统调用

**用户态 → 内核态的唯一途径。**

用户程序通过 `int 0x80`（x86）或 `syscall`（x64）指令陷入内核，OS 内核代替用户完成特权操作（读文件、创建进程等）。

> 流程：用户程序 → 陷入指令 → 用户态切内核态 → 内核处理 → 返回用户态

## 二、进程管理

### 2.1 进程定义

**进程 = 正在运行的程序** = 程序 + 数据 + PCB（进程控制块）。

PCB 是 OS 用来记录进程信息的数据结构，包含 PID、程序计数器、寄存器、内存限制、打开的文件列表等。

### 2.2 进程 vs 线程

| | 进程 | 线程 |
|------|------|------|
| 比喻 | 工厂 | 工厂里的工人 |
| 资源 | 拥有独立资源 | 共享进程的资源 |
| 调度 | 切换代价大 | 切换轻量 |
| 通信 | IPC（管道、消息队列等）| 直接读写共享变量 |

### 2.3 进程五状态

```
创建 → 就绪 → 运行 → 终止
         ↖   ↓
          阻塞
```

| 状态转换 | 触发条件 |
|----------|---------|
| 就绪 → 运行 | 被调度选中 |
| 运行 → 就绪 | 时间片用完 |
| 运行 → 阻塞 | 等 I/O、等锁、等信号 |
| 阻塞 → 就绪 | I/O 完成、锁释放 |

### 2.4 进程间通信（IPC）

| 方式 | 说明 |
|------|------|
| 管道（pipe）| 单向数据流，父子进程间常用 |
| 消息队列 | 异步发送消息，有缓冲区 |
| 共享内存 | 最快，直接访问同一块内存，但需配合锁 |
| 信号（signal）| `kill -9 PID` 发的是 SIGKILL 信号 |
| 套接字（socket）| 跨网络通信 |

### 2.5 临界区

访问共享资源的代码段叫临界区。必须保证同一时刻只有一个进程在里面。

**临界区访问准则**：
- 空闲让进
- 忙则等待
- 有限等待（不能永远等）
- 让权等待（等的时候释放 CPU）

## 三、CPU 调度

### 3.1 核心问题

CPU 只有一个（或几个核），几十个进程等着跑。**谁先用？用多久？**

### 3.2 调度算法

| 算法 | 规则 | 优缺点 | 抢占？ |
|------|------|--------|--------|
| **FCFS**（先来先服务）| 谁先到谁先跑 | 简单，但长进程堵死短进程 | 否 |
| **SJF**（短作业优先）| 服务时间最短的先跑 | 平均等待最少，但长作业可能饿死 | 均可 |
| **HRRN**（高响应比）| 响应比 = (等待+服务) / 服务 | 折中方案，谁响应比高谁跑 | 否 |
| **RR**（时间片轮转）| 每人跑一个时间片 q，轮流 | 公平，但 q 太大= FCFS，q 太小= 频繁切换 | 是 |
| **多级反馈队列**⭐ | 多级队列，高优先级时间片短 | 短作业快速完成，长作业不会被饿死 | 是 |

### 3.3 多级反馈队列详解

1. N 个队列，优先级从高到低
2. 队列 1 时间片最短（比如 1），队列 2 稍长（比如 2），最后一层用 FCFS
3. 新进程总是进队列 1
4. 没跑完的进程降到下一级队列
5. 低级队列只在高级队列没有进程时才跑

> 好处：短进程在高级队列就完成，长进程沉到低级队列但时间片长，不会被饿死。

### 3.4 计算公式

| 指标 | 公式 |
|------|------|
| 完成时间 | 进程跑完的时刻 |
| 周转时间 | 完成时间 - 到达时间 |
| 带权周转时间 | 周转时间 ÷ 服务时间 |
| 等待时间 | 周转时间 - 服务时间 |

### 3.5 典型考题

给一张表（进程名、到达时间、服务时间），用某种算法画出**甘特图**，算平均周转时间。

## 四、进程同步与 PV 操作

> ⭐ 大题必考，分值最高

### 4.1 信号量是什么

**信号量 = 一个整数计数器 + 一个等待队列。**

类比停车场：
- 有 3 个车位 → 信号量初值 = 3
- 来一辆车 → P 操作（车位-1），满了就排队
- 走一辆车 → V 操作（车位+1），有人排队就叫醒

### 4.2 P 和 V 操作

```
P(S)：S = S - 1
      如果 S < 0：把自己挂起，到 S 的等待队列排队

V(S)：S = S + 1
      如果 S ≤ 0：叫醒 S 等待队列里的第一个人
```

- P 是"申请资源"，V 是"释放资源"
- P 可能阻塞，V 不会阻塞

### 4.3 生产者-消费者 ⭐⭐

面包师做面包放货架（N 个位置），顾客从货架取。

```c
Semaphore empty = N;    // 空格子数
Semaphore full = 0;     // 已有面包数
Semaphore mutex = 1;    // 货架锁

// ============ 生产者（面包师）============
while (1) {
    做一个面包;

    P(empty);            // ① 先看有没有空格
    P(mutex);            // ② 占货架
    把面包放货架上;
    V(mutex);            // ③ 释放货架
    V(full);             // ④ 面包+1，有顾客在等就叫醒
}

// ============ 消费者（顾客）============
while (1) {
    P(full);             // ① 先看有没有面包
    P(mutex);            // ② 占货架
    从货架取面包;
    V(mutex);            // ③ 释放货架
    V(empty);            // ④ 空格+1，有面包师在等就叫醒

    吃掉面包;
}
```

> ⚠️ **关键坑**：①② 不能颠倒。先 P(mutex) 再 P(empty) → 货架满了面包师拿着锁睡觉，顾客拿不到锁没法取 → **死锁**。

> ⚠️ 两个 V 操作之间没有顺序要求。

### 4.4 读者-写者 ⭐⭐

公告栏：多个人可以同时看（读），但有人写时谁都不能看，也不能两人同时写。

```c
Semaphore rw = 1;        // 读写互斥锁
Semaphore mutex = 1;     // 保护读者计数器的锁
int readcount = 0;       // 当前有几个读者在读

// ============ 读者 ============
P(mutex);
readcount++;
if (readcount == 1)
    P(rw);               // 第一个读者：锁住公告栏，不让写
V(mutex);

// 读公告...

P(mutex);
readcount--;
if (readcount == 0)
    V(rw);               // 最后一个读者：释放，写者可以进来了
V(mutex);

// ============ 写者 ============
P(rw);                   // 等所有人（读或写）走
// 写公告...
V(rw);
```

### 4.5 哲学家就餐 ⭐

五个人围圆桌，五根筷子。每人需要同时拿到左右两根才能吃。

**死锁场景**：每人同时拿起左边筷子 → 都在等右边 → 全饿死。

**解法 1**：信号量限制最多 4 个人同时拿筷子
```c
Semaphore limit = 4;
// 每个人：
P(limit);
P(左筷子); P(右筷子);
吃饭;
V(右筷子); V(左筷子);
V(limit);
```

**解法 2**：奇数号先左后右，偶数号先右后左（破坏循环等待）

## 五、死锁

### 5.1 定义

一组进程互相等待对方持有的资源，永远无法推进。

### 5.2 四个必要条件（简答必背）

| 条件 | 含义 | 类比 |
|------|------|------|
| **互斥** | 资源一次只能给一个人 | 厕所一次只能进一人 |
| **请求并保持** | 拿着已有的，还要申请新的 | 你占着厕所还要等洗衣机 |
| **不可抢占** | 不能硬抢别人的资源 | 不能把厕所里的人踹出来 |
| **循环等待** | A 等 B、B 等 C、...、N 等 A | 形成等待环 |

**破坏任一条件就能防止死锁。**

### 5.3 银行家算法

银行放贷前先算：我借出这笔钱后，还有能力满足所有人后续的要求吗？

OS 同理：每次给进程分配资源前，先判断系统是否还在"安全状态"。

> 考题：给一张表（进程名、已分配资源、还需资源、系统可用资源），判断当前是否安全，请求是否可满足。

### 5.4 安全序列

存在一个进程序列，按照这个顺序分配资源，每个进程都能顺利完成 → 系统处于安全状态。找不到 → 不安全，可能死锁。

## 六、内存管理

### 6.1 虚拟内存

**每个进程"以为"自己独占全部内存。** OS 在背后把虚拟地址翻译成真实物理地址。

好处：
- 程序不用关心物理内存在哪
- 进程之间互相隔离，不能乱看别人的内存
- 可以运行比物理内存还大的程序（用硬盘补）

### 6.2 分页（Paging）

把内存切成固定大小的块：虚拟地址侧叫**页（page）**，物理地址侧叫**页框/块（frame）**。

大小通常是 4KB（4096 字节 = 2^12）。

### 6.3 页表

页表就是字典：**页号 → 块号**。

```
虚拟地址 = [页号 | 页内偏移]
                ↓ 查页表
物理地址 = [块号 | 页内偏移]  ← 偏移直接复制，不翻译
```

### 6.4 地址转换计算

> 给虚拟地址，页大小，页表 → 算物理地址

**步骤**：
1. 页大小 = 2^n → 页内偏移占 n 位
2. 虚地址 ÷ 页大小 → 商 = 页号，余数 = 页内偏移
3. 查页表：页号对应哪个块号
4. 物理地址 = 块号 × 页大小 + 页内偏移

**例**：页大小 4KB，虚地址 0x5A38

```
4KB = 2^12，低 12 位 = A38（偏移），高 4 位 = 5（页号）
页表：页 5 → 块 7
物理地址 = 7A38
```

### 6.5 快表（TLB）

页表在内存里，每次访问数据要查两次内存（先查页表，再取数据）→ 慢。

**TLB 是 CPU 里的小缓存**，存最近用过的几个页表项。一般命中率 90%+，大幅减少查页表开销。

### 6.6 多级页表

页表本身也很大（每个进程一个页表）。**把页表也分页**，只把用到的部分放内存。

二级页表 = 页目录（存"这页页表在哪"）→ 页表 → 物理地址。Linux 常用四级页表。

## 七、虚拟内存与页面置换

### 7.1 缺页中断

你要访问的页不在物理内存里（在硬盘上），触发**缺页中断**。OS 暂停当前进程，去硬盘把页读进来，再恢复执行。

类比：你书架上没这本书，得去图书馆借。一趟下来慢几百倍。

### 7.2 页面置换算法

> 核心问题：物理内存满了，要换一页出去，**换谁？**

| 算法 | 规则 | 特点 |
|------|------|------|
| **OPT**（最佳）| 淘汰将来最久才用到的 | 🏆 最优，但做不到（没法预知未来），只当基准 |
| **FIFO**（先进先出）| 淘汰最先进来的 | 实现简单，有 **Belady 异常**（越多页反而越缺页）|
| **LRU**（最近最久未用）| 淘汰过去最久没被访问的 | 效果好，但硬件支持代价大 |
| **CLOCK / NRU** | 给每页一个"访问位"，时钟指针扫到访问位=0 的淘汰 | 性能接近 LRU，实现简单 |

### 7.3 改进型 CLOCK

给每页加两个标记位：
- **A**（访问位）：最近被读过吗
- **M**（修改位）：内容被改过吗

扫描时优先级 **A=0, M=0 → A=0, M=1 → A=1, M=0 → A=1, M=1**。第一轮找不到就清 A 再扫。

### 7.4 典型考题

给一串页面访问序列，内存只有 N 个框。让你一步步画出每步内存里有什么，标缺页，最后算**缺页次数**和**缺页率**。

### 7.5 抖动（Thrashing）

**分给进程的页面太少，进程频繁缺页，大部分时间在等待换页而不是干活。**

就像你桌面只有巴掌大，每写一行代码都要起身去书架拿书 → 一上午啥也没干。

### 7.6 工作集（Working Set）

一个进程**当前时间段内实际需要的页面集合**。

OS 应该保证工作集在内存里。如果所有进程的工作集之和超过物理内存 → 抖动。

## 八、磁盘调度

### 8.1 问题

磁头要在盘面上来回移动找数据。**按什么顺序处理请求能让总移动距离最短？**

### 8.2 调度算法

| 算法 | 规则 | 特点 |
|------|------|------|
| **FCFS** | 按请求到达顺序 | 公平但寻道时间长 |
| **SSTF**（最短寻道）| 离当前磁头最近的处理 | 可能饿死远处请求 |
| **SCAN**（电梯算法）⭐ | 一个方向走到底，路上有请求就处理，到底折返 | 公平且效率好 |
| **C-SCAN**（循环扫描）| 只在一个方向处理，到底直接跳回起点（不处理）| 更公平 |
| **LOOK** | 类似 SCAN 但不用走到底，没请求了就折返 | 实际系统常用 |

### 8.3 典型考题

```
当前磁头位置：100
请求序列：55, 58, 39, 18, 90, 160, 150, 38, 184
```

画出磁头移动轨迹，计算总寻道长度。

## 九、文件系统

### 9.1 核心概念

| 概念 | 说明 |
|------|------|
| **inode** | 文件的"身份证"。存文件大小、位置、权限、时间戳。**不存文件名。** |
| **目录** | 本质是一个表：`文件名 → inode 号` |
| **超级块** | 存整个文件系统的元信息（总大小、空闲块数等）|
| **块** | 文件系统的最小读写单位（通常 4KB）|

### 9.2 路径解析

`/home/xiao/file.txt` 是怎么找到的：
1. 找到 `/` 的 inode → 找到数据块（目录项）
2. 在目录项里找到 `home` → 得到 home 的 inode
3. 在 home 目录里找到 `xiao` → 得到 xiao 的 inode
4. 在 xiao 目录里找到 `file.txt` → 得到文件的 inode → 读取数据块

### 9.3 硬链接 vs 软链接

| | 硬链接 | 软链接（符号链接）|
|------|--------|------------------|
| 是什么 | 多个文件名指向**同一个 inode** | 一个文件存的是**目标路径字符串** |
| 删原文件 | 不影响，其他硬链接还在 | 软链接失效（悬空）|
| 跨文件系统 | ❌ 不能（不同分区 inode 独立）| ✅ 可以 |
| 用途 | 防止误删 | 快捷方式、版本管理 |

### 9.4 常见文件系统

| 文件系统 | 特点 |
|----------|------|
| **ext4** | Linux 老牌默认，稳定可靠 |
| **btrfs** | 支持快照、压缩、写时复制（你用的）|
| **XFS** | 高性能，适合大文件 |
| **NTFS** | Windows |
| **FAT32/exFAT** | U 盘通用 |

### 9.5 文件分配方式

| 方式 | 原理 | 优缺点 |
|------|------|--------|
| **连续分配** | 文件占用连续的磁盘块 | 访问快，但会有外部碎片 |
| **链接分配** | 每个块存下一块的指针 | 没有外部碎片，但随机访问慢 |
| **索引分配** | 用一个索引块存所有数据块地址 | 综合最优，Unix/Linux 用这个 |

## 十、I/O 管理

### 10.1 I/O 控制方式演进

```
程序轮询 → 中断驱动 → DMA → 通道
  (慢)                      (快)
```

| 方式 | 谁控制 | CPU 参与度 |
|------|--------|-----------|
| **轮询** | CPU 不停检查"好了没"| CPU 全程忙等 |
| **中断** | 外设好了发中断信号 | CPU 启动传输后干别的，传输完毕被中断 |
| **DMA**（直接内存访问）| DMA 控制器 | CPU 只布置任务，DMA 自己搬数据，搬完通知 CPU |
| **通道** | 通道处理器 | 更高级的 DMA，有自己的指令集 |

### 10.2 缓冲

**为什么需要缓冲**：CPU 快，I/O 设备慢。缓冲像中转站，平滑速度差。

- **单缓冲**：一个缓冲区，处理时间和传输时间可重叠一部分
- **双缓冲**：两个缓冲区交替用，一个填数据一个处理数据，并行度更高
- **环形缓冲**：多个缓冲区首尾相连，生产者消费者模型

### 10.3 SPOOLing（假脱机）

多用户"同时"用打印机。实际上 OS 先把每个用户的打印任务存到磁盘（输出井），再逐个发给打印机。用户以为自己在独占打印机。

## 十一、启动过程

按下电源到看到桌面：

```
电源 → BIOS/UEFI 自检 → 找启动设备 → GRUB（引导菜单）
    → 加载内核（vmlinuz）→ initramfs（临时根文件系统）
    → 启动 systemd（PID=1）→ 启动服务（网络、音频、显示...）
    → getty（登录终端）或 显示管理器 → 登录 → 桌面
```

你 U 盘启动时选"猫头鹰 OS Live"，就是在 GRUB 这一步选择了不同的启动项。
